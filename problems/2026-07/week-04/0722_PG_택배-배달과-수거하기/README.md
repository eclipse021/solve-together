## 짧은 회고

### 준현

- 접근: 처음엔 완전 탐색으로 접근했으나 시간 복잡도상 불가능해서, 가장 먼 곳부터 처리하는 그리디로 방식을 PQ를 사용해 넣고 빼는 방식으로 구현했다.
- 막힌 지점: 그리디는 감이 잡혔지만 무조건 멀리서 처리해야하는 명확한 규칙을 잡는 데 오래 걸렸고, 배달/수거 배열을 동시에 관리하는 구조를 깔끔하게 짜는 데도 시간이 걸렸다.
- 다음에 기억할 것: stack 방식으로 구현할 수 있음 / 시간 복잡도 계산 올바르게 하기(반복문 수 만큼 pq.top 추가 계산해주기)

### 준희

- 접근: "가장 먼 집부터 처리하는 그리디"라는 방향은 처음부터 잡고 있었음. 뒤에서부터 순회하며 배달/수거 잔량을 각각 변수 하나(`del_sum`, `pick_sum`)에 누적하고, 둘 중 하나라도 남아 있으면 `cap`씩 빼면서 `(i+1)*2`를 더하는 방식으로 해결.
- 막힌 지점: 처음엔 반복문 안에서 매번 `sum()`으로 남은 양을 다시 계산하는 시뮬레이션으로 짰음. 방향은 맞았지만 계산을 계속 반복하는 구조라 비효율적이었고, 나중에 "배열형 누적합"이 아니라 **변수 하나에 계속 더해가며 상태를 갱신하는 변수형 누적**으로 바꿔야 한다는 걸 알게 됨.
- 다음에 기억할 것: 반복문 안에서 `sum()`을 다시 부르고 있으면 거의 항상 누적 변수로 대체 가능하다는 신호(0716 두 큐 문제와 같은 패턴으로 두 번째로 걸림). 배달/수거처럼 서로 독립적인 두 자원은 각각 누적해두고 `while a > 0 or b > 0`로 같이 소모시키면 왕복 횟수가 자연스럽게 나온다.
