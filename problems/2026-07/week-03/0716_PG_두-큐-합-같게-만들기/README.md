## 짧은 회고

### 준현

- 접근: 누적합 목록을 set에 전부 저장 후, 현재 큐의 전체합에서 set 목록 중 절반으로 만드는 누적합이 있는지 탐색하는 방식으로 문제 해결
- 막힌 지점: 큐에서는 앞에서 빼고 뒤에 넣는 연산만 가능한데, 순환 배열의 양쪽 경계를 자유롭게 선택할 수 있다고 생각해 투 포인터 방식을 생각 못함,,,
- 다음에 기억할 것: 1) 양수 조건, 2)앞 뒤로 값을 넣고 빼는 조건 나오면 투 포인터 방식 생각하기 + 큐의 삽입/삭제 방향 헷갈리지 않기!

### 준희

- 접근: "최소 이동 보장"이라 처음엔 DP/백트래킹/브루트포스로 모든 경우를 탐색한 뒤 최소 해를 찾는 방식을 떠올렸으나 시간초과가 우려됨. 20분 정도 고민 후 어떤 알고리즘을 쓸지만 힌트를 얻음 → 그리디(합이 큰 쪽 큐의 front를 빼서 반대 큐 back으로). 방법은 바로 떠오름.
- 막힌 지점: (1) 반복문 안에서 매번 `sum()`을 다시 계산해 시간초과 → 합을 처음 한 번만 구하고 이동할 때마다 증감으로 갱신. (2) 반복 횟수를 큐 길이의 2배로 뒀더니 특정 테스트 하나만 실패 → 큐 길이의 합만으로는 부족, 최대 약 3배(3n)까지 이동이 필요함을 깨닫고 3배로 늘려 통과.
- 다음에 기억할 것: 반복문 안에서 `sum()` 재계산 금물(누적 갱신). 이 문제의 이동 상한은 큐 길이의 약 3배. "최소 보장" 문제는 그리디부터 의심.
